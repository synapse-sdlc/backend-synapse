# Skill: Code Scaffold Generation

## Purpose
Generate implementation-ready code scaffold files from an approved technical plan. Each file contains function signatures, docstrings referencing specs and acceptance criteria, imports, and TODO markers — so developers or AI coding agents can start implementing immediately.

## Pre-requisites
- Approved plan artifact exists (with subtasks, affected_routes, data_flow)
- Spec artifact exists (with user stories and acceptance criteria)
- Tests artifact exists (with test cases)
- Codebase must be indexed (to match existing patterns, frameworks, naming conventions)

## Steps

1. Read the **plan** artifact using `get_artifact` to understand subtasks, affected routes, new files, and data flow.
2. Read the **spec** artifact using `get_artifact` to get user stories and acceptance criteria.
3. Read the **tests** artifact using `get_artifact` to understand test cases that need skeletons.
4. Use `search_codebase` to find:
   - Existing file patterns (how files are structured, imports used)
   - Framework conventions (React components, FastAPI routes, Express handlers, etc.)
   - Test patterns (which test framework, how tests are organized)
5. For **each plan subtask**, generate a scaffold file:
   - Match the language/framework of the existing codebase
   - Include proper imports based on existing patterns
   - Create function signatures with full docstrings referencing the subtask ID, story ID, and acceptance criteria
   - Add TODO comments with implementation hints from the plan
   - If the subtask involves a new route/endpoint, scaffold the route handler
   - If the subtask involves a migration, include the migration scaffold
6. For **each relevant test suite**, generate a test skeleton file:
   - Match the existing test framework (pytest, jest, vitest, etc.)
   - Create test function stubs for each test case
   - Include setup/teardown if existing tests use them
   - Reference the test case ID in the function name or docstring
7. Store the scaffold using `store_artifact` with type="scaffold" and parent_id=plan_artifact_id.

## Output Schema

Call `store_artifact` with type="scaffold", parent_id="<plan_artifact_id>" and content as JSON:

```json
{
  "feature_name": "...",
  "plan_id": "<plan_artifact_id>",
  "spec_id": "<spec_artifact_id>",
  "scaffold_files": [
    {
      "path": "src/services/token_service.py",
      "language": "python",
      "subtask_id": "ST-002",
      "story_id": "US-001",
      "description": "Token generation and validation service",
      "functions": ["generate_reset_token", "validate_token", "invalidate_tokens"],
      "content": "# Full file content as a string..."
    }
  ],
  "summary": {
    "total_files": 6,
    "total_functions": 14,
    "total_test_files": 3,
    "languages": ["python"],
    "subtasks_covered": ["ST-001", "ST-002"],
    "subtasks_total": 7
  }
}
```

## Guidelines

- **Every subtask** in the plan SHOULD have at least one scaffold file
- Each file MUST have a header comment with: source subtask ID, story ID, "Generated by Synapse"
- Function docstrings MUST include the relevant acceptance criteria (GIVEN/WHEN/THEN)
- Use `raise NotImplementedError("ST-XXX")` or `// TODO: Implement ST-XXX` as placeholder bodies
- Match the existing codebase style — if it uses classes, use classes; if functional, use functions
- Import statements should reference actual modules found in the codebase
- Test files should follow existing test naming conventions (test_*.py, *.test.js, *.spec.ts)
- Include a `summary` field counting total files, functions, test files, and subtask coverage
